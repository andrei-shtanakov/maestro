# Gates v1.2 — pipeline-position resume (H-6) + harness-artifact isolation (H-7)

**Date:** 2026-07-13
**Status:** approved (design review 2026-07-13)
**Findings:** H-6, H-7 from governed run #2
(`../../../../prograph-vault/derived/journal/Maestro/journal.md`, 2026-07-12 23:05)
**Prior art:** gates v1.1 (H-3..H-5, PR #75), WS-006 M-1..M-3 (PR #73)

## Problem

### H-6 — ex-post approval is destroyed by the resume path

An ex-post gate block routes `RUNNING -> FAILED -> NEEDS_REVIEW`; the operator
re-queues via `maestro workstream-approve` (`NEEDS_REVIEW -> READY`, preserving
`error_message` with the marker `gates:approval-required phase=ex_post sha=X`).
The main loop then calls `_spawn_workstream`, which unconditionally replays the
whole pipeline: `READY -> DECOMPOSING` -> spec regeneration (LLM,
non-deterministic) -> new spec commit -> new worktree HEAD -> full
`spec-runner run --all` re-run -> ex-post re-evaluated at a *new* sha. The
approval was bound to the old sha (DESIGN-608), so the workstream blocks again.
The loop never converges; in run #2 the operator finished by hand.

### H-7 — orchestrator `spec/` collides with the target repo's own `spec/`

steward keeps its own governance docs in `spec/` (00-charter.md, …). Maestro
generates `spec/{tasks,requirements,design}.md`, `.executor-state.db`, logs
into the same directory and commits `spec/` + `spec-runner.config.yaml` into
the feature branch — harness artifacts ride into the target repo's PR, and a
name collision would silently clobber target-owned files. Worse, the H-4
exclusion `_ORCHESTRATOR_MANAGED = ("spec/", "spec-runner.config.yaml")`
hides from ex-post *any* change under `spec/` — including agent edits to the
target repo's real spec docs.

## Design

No new DB status (`AWAITING_MERGE` rejected — enum/schema/dashboard churn for
the same outcome). No spec-runner changes: `spec_prefix` is an existing,
supported mechanism (multi-phase support).

### H-6: resume-in-place

**Entry point.** The marker check is the FIRST action in `_spawn_workstream`
after reading the workstream — before the `DECOMPOSING` transition and before
the `_SPAWNING_SENTINEL` write (orchestrator.py:539). A new helper
`gates.parse_approval_marker(error_message) -> ApprovalMarker | None` parses
`phase` and `sha` out of the stored block reason.

**Resume condition.** Marker present with `phase=ex_post` AND the worktree for
this workstream exists AND `git rev-parse HEAD` in the worktree equals the
marker's sha.

**Resume path.**

1. Transition `READY -> RUNNING` with explicit `process_pid=None`,
   `generation_pid=None`, `workspace_path=str(workspace)` — never inherit a
   stale PID (startup recovery would treat it as a possible live orphan).
2. Call the shared success continuation directly (no subprocess spawn).
   Ex-post re-evaluates over the unchanged worktree; the approval memory
   accepts it — the `error_message` marker survives orchestrator restarts (it
   lives in the DB), unlike the verdict store bound to the run's
   `logs/<ULID>/` directory. Then the normal tail: `MERGING` -> PR ->
   merge-into-base -> `DONE`.
3. Clear `error_message` explicitly on the `DONE` write — and ONLY there.
   Storage never clears the field on its own (database.py:1842), so the clear
   needs an explicit mechanism in `update_workstream_status` (a plain `None`
   kwarg means "leave untouched" today). Clearing earlier (e.g. at `MERGING`)
   would break the crash-tail: startup recovery resets a stranded
   MERGING/PR_CREATED workstream to READY (orchestrator.py:200), and with the
   marker already gone the next run would full-respawn instead of resuming.
   Keeping the marker until DONE makes the whole post-approval tail
   re-enterable: crash anywhere in MERGING/PR/merge -> recovery -> READY ->
   marker + unchanged HEAD -> resume-in-place again (PR creation already
   tolerates an existing PR).

**Refactor.** `_handle_success(workstream_id, running: RunningWorkstream)`
uses only `workspace_path` from the running record; change the signature to
`(workstream_id, workspace_path: Path)` (same for `_gate_ex_post`). Callers:
`_handle_completion` passes `running.workspace_path`; the resume path passes
the worktree path from the workspace manager / DB.

**Fallback.** Worktree missing or HEAD != marker sha -> log a warning and take
the existing full-respawn path. The approval is genuinely invalid
(DESIGN-608), so a fresh run is correct. A `phase=ex_ante` marker changes
nothing: ex-ante sha is the *main repo* HEAD, which a respawn does not move,
so ex-ante approvals already survive the replay.

**Stats.** The ex-post block counted `failed += 1`; a successful resume counts
`completed += 1` as usual. No change.

### H-7: prefix + ignore, no harness commits

1. **`spec_prefix = "maestro-"`** (module constant). Lands in the generated
   config's `executor:` section (spec-runner reads `spec_prefix` from the
   wrapper) AND is passed explicitly as `--spec-prefix maestro-` to both
   `spec-runner plan --full` and `spec-runner run --all` (belt and braces —
   the CLI default `""` is falsy and must not override the config).
   Artifacts become `spec/maestro-{tasks,requirements,design}.md`,
   `spec/.executor-maestro-state.db`, `spec/.executor-maestro-logs`: no name
   collision with target-owned files.
2. **Order change in `_spawn_workstream`:** `setup_spec_runner` (config
   write) moves BEFORE `generate_spec`, so `plan --full` writes prefixed
   paths from the start.
3. **Prefix-aware pass over every hardcoded name:**
   - `decomposer.generate_spec` post-check: `spec/maestro-tasks.md` (prefix
     passed in as a parameter; today it checks `spec/tasks.md`,
     decomposer.py:286).
   - `spec_runner.read_executor_state` (progress reader, spec_runner.py:41):
     must read `.executor-maestro-state.db` / `.executor-maestro-state.json`
     — `read_executor_state` gains a `prefix: str = ""` parameter applied to
     `SQLITE_STATE_FILENAME`/`JSON_STATE_FILENAME`. Without this, progress
     reads empty forever. Caller: orchestrator.py:806.
   - `workspace.setup_spec_runner` stale-state cleanup: prefixed filenames.
4. **Ignore instead of commit.** `_commit_spec_in_workspace` is deleted.
   Instead, an idempotent marker block is appended once to the target repo's
   `$GIT_COMMON_DIR/info/exclude` (repo-local, shared by all linked worktrees
   — git resolves a linked worktree's `info/exclude` to the common dir; this
   is NOT per-worktree, and the spec says so deliberately). Patterns are
   narrow:

   ```
   # >>> maestro orchestrator artifacts (auto-managed) >>>
   spec/maestro-*
   spec/.executor-*
   /spec-runner.config.yaml
   # <<< maestro orchestrator artifacts <<<
   ```

   Ignored files survive spec-runner's subtask branching/merges (untracked
   files are untouched by checkout/merge) and are invisible to the agent's
   `git add -A` auto-commit (hooks.py:375 respects gitignore). No harness
   artifact reaches the feature branch or the PR. `spec/.executor-*` is
   deliberately broader than the prefix: runtime state (incl. the unprefixed
   `.executor-stop`) must never be committed by anyone.
5. **Gates exclusion narrowed.** `_ORCHESTRATOR_MANAGED` becomes
   `("spec/maestro-", "spec/.executor-")` (prefix match). Changes to the
   target repo's real `spec/*.md` are ex-post-visible again.
   `spec-runner.config.yaml` is REMOVED from the exclusion: in a repo that
   tracks its own config, Maestro's overwrite shows up in the diff and
   triggers `scope_violation` — a desired safety net; in a repo without one
   the file is untracked+ignored and never enters the diff.
6. **Prefix-contract guard.** spec-runner is an external subprocess, not a
   dependency (pyproject.toml pins nothing at runtime) — an old spec-runner
   without `--spec-prefix` would break H-7 silently (artifacts land at
   unprefixed paths; the ignore block and the narrowed gates exclusion both
   miss them). Preflight gains a check that the installed spec-runner
   advertises `--spec-prefix` (`spec-runner run --help` output); failure is a
   preflight ERROR, not a warning — gates + prefix isolation must not run on
   a contract the binary does not support.
7. **Preflight warning (M1) — fail-closed candidate.** If the target repo
   TRACKS its own `spec-runner.config.yaml`, warn: Maestro will overwrite a
   target-owned file in the worktree. This is a real ownership conflict, not
   just noise. M1 ships warning + the ex-post scope_violation backstop from
   (5); a later hardening step should turn this into a hard preflight error
   unless spec-runner grows an alternative config path.

**Interaction bonus.** With no spec commit, a respawn no longer moves the
worktree sha by itself — fewer spurious approval invalidations feeding H-6's
fallback path.

## Testing

- Unit: `parse_approval_marker` (round-trip with the blocked_reason format,
  garbage input, ex_ante vs ex_post).
- Orchestrator (mocked spawn):
  - resume: READY + ex_post marker + matching HEAD -> no DECOMPOSING, no
    subprocess, ex-post re-eval -> MERGING -> DONE; `error_message` cleared
    on DONE (and still present at MERGING/PR_CREATED);
    `process_pid`/`generation_pid` are None throughout.
  - crash-tail re-entry: workstream reset to READY from stranded
    MERGING/PR_CREATED with marker intact -> resume-in-place fires again.
  - fallback: sha mismatch -> warning + full respawn; missing worktree ->
    full respawn; ex_ante marker -> full respawn.
- H-7:
  - config written before `plan --full` (call-order assertion);
  - `--spec-prefix maestro-` present on plan and run argv;
  - decomposer checks `spec/maestro-tasks.md`;
  - progress reader finds `.executor-maestro-state.db`;
  - exclude block idempotent (double setup -> single block);
  - `_ORCHESTRATOR_MANAGED`: `spec/maestro-tasks.md` and
    `spec/.executor-maestro-state.db` excluded, `spec/00-charter.md` and
    `spec-runner.config.yaml` NOT excluded;
  - preflight warning fires on a tracked `spec-runner.config.yaml`;
  - preflight ERROR when `spec-runner run --help` lacks `--spec-prefix`
    (prefix-contract guard; mocked help output in unit tests).
- Regression: existing gates v1.1 tests stay green (approval memory,
  H-3 marker semantics unchanged).

## Out of scope

- Hard preflight failure on tracked `spec-runner.config.yaml` (fail-closed
  candidate, needs an alternative config path in spec-runner first).
- spec-runner changes of any kind (H-1/H-2 remain spec-runner-side handoffs).
- New workstream statuses or DB schema changes.
