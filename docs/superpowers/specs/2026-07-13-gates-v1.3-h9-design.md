# Gates v1.3 — durable approval memory + audit trail (H-9)

**Date:** 2026-07-13
**Status:** approved (design review 2026-07-13)
**Finding:** H-9 (P1) from governed run #3
(`prograph-vault/authored/notes/2026-07-13-governed-run-3-findings.md`)
**Prior art:** gates v1.1 (H-3, approval memory via verdict-store), v1.2 (H-6
resume-in-place, DONE-only marker clear), v1.2.1 (H-8).

## Problem

Approval authority today is split across two fragile carriers:

1. **`error_message` marker** (`gates:approval-required phase=<p> sha=<sha>`)
   — a single-slot text field with at least three writers that destroy it:
   `_handle_failure` (observed live in run #3: a genuine `plan --full` failure
   between the operator's approval and the gate evaluation overwrote the
   marker), the shutdown write (`"Orchestrator shutdown"`), and
   `_route_gate_block` itself (a block in one phase replaces the other
   phase's marker — the H-3 ghost).
2. **Verdict-store lookup** (`_prior_block_recorded`) — reads only the
   CURRENT pipeline's `logs/<ULID>/gate_verdicts.jsonl`; every `maestro
   orchestrate` invocation mints a new ULID, so the memory does not survive
   restarts. In run #3 this combination lost a real human approval and
   re-blocked on the same sha.

Root cause: approval is *inferred* from transport artifacts (a message
string, telemetry records) instead of being *recorded* as the explicit act
it is.

## Design

**Authority moves to the DB, recorded at the moment of the approval act.**
H-9 is the next layer above H-6: `error_message` and the verdict-store
remain transport and evidence respectively — never a source of truth.

### Schema — table `gate_approvals`

```sql
CREATE TABLE IF NOT EXISTS gate_approvals (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    workstream_id TEXT NOT NULL,
    phase TEXT NOT NULL CHECK (phase IN ('ex_ante', 'ex_post')),
    sha TEXT NOT NULL,
    approved_at TEXT NOT NULL,
    UNIQUE (workstream_id, phase, sha)
)
```

- Added BOTH to `SCHEMA_SQL` (fresh databases get it immediately) AND as
  **migration v6** in the linear journal (`CREATE TABLE IF NOT EXISTS ...`,
  idempotent for existing databases) — the established dual pattern.
- Append-only; `INSERT OR IGNORE` makes re-approval idempotent
  (UNIQUE(workstream_id, phase, sha)).
- Nothing ever deletes rows: DONE does not clear them (historical audit),
  and approvals for superseded shas are inert by construction.
- `approved_by` is deliberately deferred. This is **durable approval memory
  + audit trail** (sanctioned CLI point + timestamp), not a full human-audit
  system; the column can be added later without redesign.

### Database API

- `record_gate_approval(workstream_id: str, phase: str, sha: str) -> None`
  — `INSERT OR IGNORE`, `approved_at = now(UTC)`.
- `list_gate_approvals(workstream_id: str) -> set[tuple[str, str]]`
  — `(phase, sha)` pairs for gate evaluation.

### The approval act — `workstream-approve` in one transaction

`_approve_workstream` (maestro/cli.py) becomes: parse the marker from
`error_message` → within a SINGLE transaction: `INSERT OR IGNORE` into
`gate_approvals` (when a marker is present) + status flip
`NEEDS_REVIEW -> READY`. The transaction removes the intermediate state
"approval recorded but the row still NEEDS_REVIEW" outright (the two-step
crash analysis was safe in both directions, but atomic is simply better).

The transaction CANNOT be composed naively from the current DB API:
`update_workstream_status()` commits per call (database.py:1842) and the
transaction context manages commit/rollback separately (database.py:590).
The design therefore adds a dedicated DB method —
`approve_workstream_with_gate_record(workstream_id, phase: str | None,
sha: str | None) -> Workstream` — that executes the `INSERT OR IGNORE`
(when phase/sha are given) and the guarded `UPDATE workstreams SET
status='ready' WHERE id=? AND status='needs_review'` (rowcount-checked)
on one connection inside one transaction, without calling the
commit-heavy helper internally. `phase=None` → status flip only (the
no-marker requeue).

- Marker present → record + requeue; print what was approved:
  `approved phase=ex_post sha=6b7c8d5 (recorded)`.
- No marker → requeue only, with an explicit note: `no gate marker in
  error_message — requeued WITHOUT recording an approval`.

This is the only sanctioned approval point. A raw
`sqlite3 "UPDATE workstreams SET status='ready'"` no longer grants an
approval — intentionally fail-closed.

### GateKeeper — approvals as input, single source

- `evaluate_ex_ante` / `evaluate_ex_post` replace the `prior_error`
  parameter with `approvals: set[tuple[str, str]]`; the orchestrator loads
  `list_gate_approvals(workstream_id)` before each evaluation and passes it.
- `_decide`: `approved = (phase, sha) in approvals`. Nothing else grants
  approval: the marker-in-`prior_error` check is REMOVED as an authority
  source, and `_prior_block_recorded` is DELETED — the verdict-store returns
  to being pure telemetry/evidence (EvidenceRef).
- The marker keeps its two non-authority roles unchanged: operator-facing
  UX in the block message (it tells the operator what to approve and is
  what `workstream-approve` parses), and the H-6 pipeline-position signal
  (`_try_resume_ex_post` trigger). The resume trigger already ignores
  `phase=ex_ante` markers and continues to.

### Degradation semantics (sha-bound, DESIGN-608 wins)

**DB approval survives marker loss, but remains sha-bound.** If a failure
or shutdown destroys the position marker after an ex-post approval, the
workstream falls back to a full respawn — and the eventual gate evaluation
passes via the DB record ONLY when the worktree HEAD still equals the
approved sha. After H-7 that is the common case (no harness commit moves
HEAD), but when a respawn genuinely mints a new commit, DESIGN-608 must
win: the old approval row is inert and the gate blocks again for a fresh
human decision. This is the intended trade — durability against transport
loss, never against content change.

### Secondary hardening — marker preservation in `_handle_failure`

Not authority (that is the DB now) but H-6 position retention: losing the
marker to a failure message costs a wasteful full respawn. `_handle_failure`
adopts the PR-note append pattern via a small helper that is **idempotent**
(extracts an existing marker and re-appends it once — never duplicates):
`error_message = f"{failure} | {marker}"` when the previous message carried
a marker. The `ex_post` marker is the one that matters for resume;
`ex_ante` markers may be preserved the same way, but the resume trigger
ignores them regardless.

**Recovery invariant change (REQUIRED — v1.2's rule becomes wrong here).**
v1.2's startup recovery routes any FAILED row carrying a marker to
NEEDS_REVIEW (orchestrator.py:311), on the premise that FAILED+marker can
only mean an interrupted gate-block two-write. Once `_handle_failure`
preserves markers, that premise breaks: an ordinary retryable failure with
an appended marker, caught by a crash between FAILED and READY, would be
wrongly parked in NEEDS_REVIEW. The recovery predicate changes from
"marker present" to "the row's `error_message` IS a gate-block reason":
`error_message.startswith(BLOCK_REASON_PREFIX)` where
`BLOCK_REASON_PREFIX = "gates: human.owner_approval required"` is exported
from `maestro/gates.py` (the same literal `_decide` builds the blocked
reason from). A failure message with an appended marker
(`"spec-runner ... | gates:approval-required ..."`) does not match the
prefix and follows the normal retry-rule recovery.

### Back-compat / migration notes

- Workstreams approved under the old semantics (marker/verdict-store) but
  not yet past the gate at upgrade time will block once more; a repeated
  `workstream-approve` records the approval durably. One-time operator note.
- No changes to the verdict-record format, EvidenceRef, or the workstream
  state machine.

## Testing

- **H-9 end-to-end regression:** block → approve (DB row written) → a
  genuine failure overwrites `error_message` → retry reaches the gate →
  passes via the DB record. (The exact run-#3 arc.)
- **Cross-phase arc:** ex-ante approval survives an ex-post block message
  and an orchestrator restart (new pipeline ULID) — passes ex-ante on
  re-evaluation.
- **Negative authority test (key regression guard):** a marker present in
  `prior_error`-style `error_message` WITHOUT a DB row does NOT grant
  approval — the gate blocks.
- **sha-bound:** DB approval for sha X does not approve an evaluation at
  sha Y (DESIGN-608).
- DB: `record_gate_approval` idempotent (double insert → one row);
  `list_gate_approvals` returns the `(phase, sha)` set; migration v6
  idempotent on both fresh and journal-bearing databases.
- CLI: approve with marker records + requeues atomically; approve without
  marker requeues with the warning and records nothing.
- Marker-preservation helper: appends once, never duplicates on repeated
  failures; H-6 resume still fires after a failure that rewrote
  `error_message` (position retained).
- **Recovery edge (the main new one):** approve → genuine failure with the
  marker preserved (appended) → crash before the READY write → startup
  recovery reconciles to READY (retry rule), NOT NEEDS_REVIEW — the
  failure-with-appended-marker does not match `BLOCK_REASON_PREFIX`.
- Recovery still parks a true interrupted gate block: FAILED row whose
  `error_message` starts with `BLOCK_REASON_PREFIX` → NEEDS_REVIEW
  (v1.2 behavior preserved under the new predicate).
- Existing gates v1.2/v1.2.1 suites stay green (H-6 resume trigger and
  DONE-only clear untouched).

## Out of scope

- `approved_by` attribution and any UI over `gate_approvals` (dashboard
  read view is a natural follow-up).
- REST/MCP approval endpoints (CLI remains the single sanctioned point).
- spec-runner-side findings H-1/H-2.
